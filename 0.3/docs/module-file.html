测试模块

所有测试代码全部放在一个文件中显然不是一种好选择。htte 允许测试代码分散到多个文件中。每个文件被 Htte 加载后都成为了模块。
通常同一业务对象接口测试放到一个模块。比如将文章增删改查接口测试可以全部归到一个模块 article 中。

依赖属性

依赖有两个属性:

  • name: 依赖名
    模块名缺省时通过模块文件路径生成。
    模块名用于跨模块数据引用。
  • module: 依赖模块的文件路径
    推荐使用相对路径（注意：路径相对于当前模块所在目录）

怎么知道依赖关系？

  • 如果模块 B 引用了模块 A 的数据（如登录 token)，则模块 B 必然依赖模块 A。
  • 如果模块 B 必须在模块 A 之后执行，则模块 B 依赖模块 A。

依赖与模块执行顺序

Htte 加载 rootDir 下的所有测试文件并生成模块。模块的执行顺序并没有设置，Htte 是如何决定哪些先执行哪些后执行的呢？

首先 Htte 通过模块的文件路径进行一道排序，然后根据模块之间的依赖再进行一次排序，保证被依赖的模块总是在前。

这种策略意味着 Htte 是不允许循环依赖的，实际上也不应该出现彼此依赖这种情况。

测试集

测试集可以包含多个测试单元和多个子测试集，每个子测试集又可以包含测试单元和其它测试集。它形成树状结构。

测试集有两个属性:

  • describe: 描述测试集测试目的
  • units: 子测试数组

测试集存在的意义是归组。有时候测试某个功能可能需要调用多个接口，就可以使用测试集组一下。

测试单元代表单个接口测试。

属性

  • describe: 测试目的
  • api: 接口对象，如果为字符串则引用的 配置属性apis 中的接口对象
  • name: 测试名
  • req: 请求数据描述
    • params: 请求路径参数，例如接口属性 uri 若为 /articles/{slug}/comments/{id}, 则有路径参数 slug 和 id
    • headers: 请求头
    • query: querystring
    • body: 请求数据，Htte 会根据接口属性 type 解压成具体格式，例如 type 为 json，则请求数据会以 json 格式 {"content": "go! go! go!"} 传给接口服务
  • res: 预期响应数据，Htte 将会将实际响应数据与该预期响应数据比对，判断测试是否通过
    • status: HTTP 状态码
    • headers: 响应头
    • body: 响应数据，Htte 会根据实际响应头 Content-Type 将实际响应数据解码，并与此处预期响应数据比对

执行过程

Htte 通过 api 去配置中查找接口描述

Htte 根据测试单元属性 req 构造请求

请求发送给后端服务器

将实际收到的响应数据和预期响应数据 res 对比，这个过程中会进行如下校验:

  • 响应状态码 200
  • 响应头中包含字段 Content-Length, 且值为 26
  • 响应体中是一个对象，该对象有一个字段 json, 其值为也是一个对象为 {content: 'go! go! go!'}

如果校验全部通过，则测试通过，有一条失败，则测试失败。
